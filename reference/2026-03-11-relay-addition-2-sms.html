Relay's web app is where you manage your pipeline. But the moment you actually need context — walking to a meeting, in a cab, between calls — you don't have the app open. You have your phone. What Envoy got right — and what Relay does differently
Envoy's core insight was correct: founders need to query their pipeline from wherever they are, in natural language, without opening an app. But the architecture Envoy chose created problems Relay avoids.
Envoy's approach
WhatsApp-native agent
  • WhatsApp via web emulation — Meta can break it without notice
  • Full agent with no scope discipline — "focus constraints as pricing lever"
  • One server per founder = linear infrastructure cost
  • No data model = intelligence claims unverifiable
  • No cold start answer — starts from zero
Relay's approach
SMS/iMessage retrieval layer
  • SMS via Twilio — 100% platform reliability, no third-party risk
  • Narrowly scoped to Output C: retrieval + Y/N on pre-queued actions
  • Shared number, LLM per message — cost: ~₹0.65/message at scale
  • Reads directly from the existing data model — same intelligence as the app
  • Cold start solved by the core MVP — SMS accesses what's already there
The moments where this matters
On-demand retrieval in the web app is correct design for the desk. SMS is correct design for everywhere else.

6
"Book me a meeting with Priya" — out of scope
Out of scope
This is Envoy's failure mode — no scope discipline meant the agent tried to do everything and became unpredictable. Relay's SMS interface has hard scope limits encoded at the LLM system prompt level. When something is out of scope, Relay says so directly — and redirects to something useful.
What you text
Book me a meeting with Priya next Tuesday
What Relay replies
Relay · 1 second
I can't book meetings from here — try Calendly or your calendar app for that. What I can do: you have an open follow-up with Priya (18d silent, cooling). Want me to surface the draft? Reply Y.
Why this scope boundary matters
The LLM system prompt for SMS responses explicitly lists what is and isn't in scope. Calendar management, email composition from scratch, new deal creation, and general questions are all blocked. The rejection is never a dead end — it always redirects to something Relay can do. This is what Envoy didn't build.

Exact scope — what SMS can and can't do
This is how Relay avoids Envoy's scope creep problem. The boundary is encoded at the system prompt level, not enforced by convention.

SMS interface scope boundary

Can do via SMS
  • Deal/contact status query (any contact by name or company)
  • "What did [name] ask me to send?"
  • "Who should I follow up with today?"
  • "What's in my pipeline right now?"
  • "When did I last talk to [name]?"
  • Snooze a follow-up (default 2d, or specify days)
  • Mark an action as completed
  • Confirm a queued draft send (Y/N)
  • Confirm a stage change suggestion (Y/N)
Cannot do via SMS
  • Book or cancel calendar meetings
  • Compose a new email from scratch
  • Create a new deal or contact
  • Answer general business questions
  • Write or edit pipeline notes
  • Access Notion directly
  • Change deal stage without Pending approval
  • Send attachments or files
